# PLAN — TG Service (аудит + работы на ~2ч)

> Дата: 2026-06-02. Метод аудита: запущенный dev-сервер (uvicorn :8001), живой прогон
> всех 25 экранов SPA через headless Chrome (playwright) + прозвон всех основных API
> эндпоинтов с admin-токеном + проверка create/start-флоу. Не по коду — вживую.

## ЭТАП 1 — Результаты аудита

### Главный вывод

UI и CRUD (создание/чтение/редактирование сущностей) **работают во всех разделах** —
ни один экран не падает с JS/console/network-ошибкой, все основные `GET` отдают `200`,
`POST .../create` отдают `201`. НО:

**🔴 Движок фоновых задач не работает локально.** Redis не установлен в системе
(`redis-server` отсутствует, brew/pm2 нет), процесс Celery-воркера не запущен
(в `ps` только uvicorn-api). При этом каждый эндпоинт `/start` переводит задачу в
`ACTIVE`/`RUNNING`, пытается `celery_app.send_task(...)`, **глушит исключение**
(`log.warning("celery_dispatch_skipped")`) и возвращает успех. Итог: пользователь
жмёт «Старт», задача показывается активной, но **никто её не исполняет** и прогресса нет.
Это делает «частично нерабочими» ВСЕ разделы с фоновым исполнением (парсер, прогрев,
рассылки, накрутка, комментинг, AI-агенты, инвайтинг, клонер, конвертер).

Воркеры при этом **написаны и реальны** (не заглушки): `commenting_tasks.py` — 691 стр.,
`ai_sales_tasks.py` — 1032 стр. (заметки в DEBT-PLAN об «недописанных воркерах» устарели).
Проблема не в коде воркеров, а в том, что их **нечем запустить** (нет брокера+воркера).

Проверено: Celery 5.4 есть в venv; kombu 5.6 есть → **filesystem-брокер Celery работает
локально без Redis вообще** (проверил end-to-end: задача исполнилась воркером). Это и есть
путь поднять движок локально.
