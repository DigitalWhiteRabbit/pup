## 🟡 Среднее (профилактика бана / удобство)

### 4. ✅ ИСПРАВЛЕНО — Гейт «человечности» перед вступлением

- Было: «голые» аккаунты (нет аватара/username) триггерят антиспам/авто-бан при join.
- **Стало:** `POST /join-chats` перед созданием задачи зовёт `_assess_humanity()` — «голый» = нет аватара ИЛИ нет username (twofa игнорируется, он невидим антиспаму), либо аккаунт не проверялся. - **Фронт:** обработчик join делает свой fetch, ловит 409, показывает `confirm()` со списком рисковых («нет аватара, нет username…») + советом заполнить профиль/прогреть → при согласии пересылает с `force_low_humanity:true`, при отмене — тихо отменяет.
- **Проверено live:** Davidhally (0 видимых маркеров) без force → 409 со списком; force обходит. JS OK.

- Было: self-reschedule loop умирал при рестарте worker (терялась pending ETA-задача) или если цикл бросал исключение.
- **Стало (3 механизма):**
  1. `pup_tg.ai_agent_reaper` (beat_schedule.py): находит ACTIVE-персоны, чей loop замолчал >2× интервала (мин 10 мин, через `julianday`), минтит свежий `loop_token` и пере-диспатчит. Свежий токен = старый/висящий цикл умирает по несовпадению → без дублей.
  2. **На `worker_ready`** (celery_app.py): reaper(`force=True`) через 10с после старта воркера — оживляет ВСЕ ACTIVE-персоны безусловно (после рестарта loop мёртв, но активность «свежая» — порог бы обманул). **Это решает основной кейс «loop переживает рестарт» БЕЗ отдельного beat-процесса.**
  3. Обёртка `ai_agent`: исключение цикла больше не убивает loop — ловится, reschedule всё равно срабатывает.
- Beat-расписание: `ai-agent-reaper-every-5-min` (300с) — периодическая подстраховка. **Beat ТЕПЕРЬ ЗАПУЩЕН** (локально + в проде через PM2 `pup-tg-beat`), активны: heartbeat, warmup_check, reaper-5min, membership-check-3h. worker_ready по-прежнему покрывает рестарты независимо от beat.
- **Проверено live:** 2× рестарт воркера → `worker_ready_reaper_dispatched` → `reaper_revived forced=True` → токен сменился → `ai_agent_rescheduled`. Loop оживает за ~10с.
