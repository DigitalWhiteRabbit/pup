## HIGH

### H1. Дефолтный admin-токен + CORS `*` = открытый API

**Где:** `app/config.py:25` (`admin_token = "replace-with-secure-token"`), `app/main.py:67-72` (`allow_origins=["*"]`), `app/core/security.py:54` (`verify_admin_token`).
**Проблема:** если в `.env` не переопределён `ADMIN_TOKEN`, единственная защита мутирующих эндпоинтов — публично известная строка из репозитория. CORS открыт для любого origin. Любая вкладка/сайт сможет дёргать API.
**Действие:**

- Fail-closed: при `environment != development` (или всегда) отказываться стартовать, если `admin_token` пуст/равен дефолту, а `pup_secret` не задан.
- Сузить `allow_origins` до реального origin ПУПа (домен/локалхост), а не `*`.
- `verify_admin_token` уже использует `hmac.compare_digest` — это хорошо, оставить.

### H2. Воркеры используют кэшированное SQLite-соединение из нескольких потоков

**Где:** `app/tasks/parsing_tasks.py:610` (`db = get_db(workspace_id)`), определение `get_db` — `app/core/database.py:1072` («cached, long-lived connection»). Celery worker запущен с `--concurrency=4` (`ecosystem.config.js`).
**Проблема:** одно долгоживущее `sqlite3.Connection` шарится между параллельными задачами/потоками одного workspace → `SQLite objects created in a thread...`, гонки на запись, блокировки. API-сторона эту проблему уже решила (per-request `connect_db` в `app/deps.py:46`), а воркеры — нет.
**Действие:** в Celery-задачах открывать **свежее** соединение на задачу (как `connect_db`) и закрывать в `finally`; либо гарантировать соединение-на-поток. Проверить все `*_tasks.py`, не только парсер.

### H3. Pause/Cancel не влияют на уже идущий парсинг

**Где:** `app/api/v1/parser.py:252` (`pause_task` ставит `PAUSED`), `app/api/v1/parser.py:290` (`cancel_task`), воркер `app/tasks/parsing_tasks.py:619` (статус читается ОДИН раз на старте) и цикл по источникам `:728`.
**Проблема:** воркер проверяет статус только в начале. Нажатие Pause не останавливает текущий прогон — он доходит до конца. Cancel полагается на `celery revoke(terminate=True)`, который жёстко убивает поток посреди `await` (грязно, задача может остаться `RUNNING`).
**Действие:** в цикле по источникам (после каждого источника, ~`:818`) перечитывать статус задачи из БД; если `PAUSED`/`CANCELLED` — корректно прерваться, сохранить прогресс/частичную аудиторию, выставить финальный статус.

### H4. Задача помечается RUNNING даже если Celery-dispatch не удался

**Где:** `app/api/v1/parser.py:220-243` (`except ... celery_dispatch_skipped` → «Continue even if Celery is not available» → всё равно UPDATE status=RUNNING).
**Проблема:** если брокер/воркер недоступен, задача навсегда зависает в `RUNNING` без исполнителя и без восстановления.
**Действие:** при ошибке dispatch — не переводить в `RUNNING` (оставить `PENDING` + вернуть понятную ошибку), либо ставить `FAILED`. Добавить reaper-добивание «протухших» `RUNNING` (в проекте reaper уже есть для loop — переиспользовать паттерн).

## MED

### M1. По одному `get_entity` на каждого пользователя — медленно и рискованно

**Где:** `parsing_tasks.py` — WRITERS `:300`, COMMENTERS `:264`, JOINERS `:439`, TOPICS `:481`.
**Проблема:** отдельный API-запрос на каждого уникального юзера. На большом чате это сотни/тысячи запросов → медленно и быстрый FloodWait/риск бана.
**Действие:** использовать объект отправителя, который Telethon уже отдаёт вместе с сообщением (`msg.sender` / `await msg.get_sender()` без отдельного resolve), либо собирать id и резолвить пачкой. Убрать поштучный `get_entity`.

### M2. `get_participants(aggressive=True)` для CHAT_MEMBERS

**Где:** `parsing_tasks.py:243`.
**Проблема:** агрессивный сбор участников — рискованно по ToS и всё равно упирается в ~10k для крупных групп.
**Действие:** сделать `aggressive` опциональным (из config), документировать лимит ~10k, для крупных групп предупреждать в UI.

### M3. COMMENTERS читает реплаи у канала, а не у привязанного чата обсуждений

**Где:** `parsing_tasks.py:258-261` (`iter_messages(entity, reply_to=msg.id)` по entity-каналу).
**Проблема:** у broadcast-каналов комментарии живут в **привязанной группе обсуждений**, а не в самом канале → reply_to по каналу может не вернуть ничего.
**Действие:** резолвить linked discussion chat (`GetFullChannelRequest` → `full_chat.linked_chat_id`) и итерировать реплаи там.

### M4. Один аккаунт на всю задачу, без ротации и без блокировки

**Где:** `parsing_tasks.py:107` (`_pick_account`), вызывается один раз `:667`.
**Проблема:** весь многосорсный парсинг идёт с одного аккаунта (рост риска бана), и две задачи могут параллельно схватить один и тот же аккаунт.
**Действие:** ротация аккаунта по источникам; помечать аккаунт «занят»/брать advisory-lock на время задачи.

### M5. Хардкод-лимиты, не выведены в конфиг

**Где:** `parsing_tasks.py` — WRITERS `limit=500` `:297`, COMMENTERS `100×50` `:258/:261`, REACTIONS `:329`, JOINERS `:424`, POLLS `:379`. `ParsingTaskConfig` (`parser.py:38`) поля лимита не имеет.
**Действие:** добавить в `ParsingTaskConfig` поля глубины/лимита и пробрасывать в хендлеры.

### M6. Нет прогресса внутри одного источника

**Где:** `parsing_tasks.py:818-825` (прогресс только после каждого источника).
**Проблема:** долгий парс одного большого источника висит на 0% до конца.
**Действие:** периодический колбэк прогресса внутри хендлера (например каждые N юзеров).

## LOW / полировка

- **L1.** `about` всегда `None` (`parsing_tasks.py:59`), `phone` почти всегда `None` (приватность) → фильтр по «about» невозможен. Документировать или убрать фильтр, либо точечный `GetFullUser`. (`_apply_filters` — `:63`.)
- **L2.** Расшифрованная сессия пишется на диск во временную папку (`parsing_tasks.py:701-703`). Чистится в `finally`, но при жёстком крэше может остаться. Лучше `StringSession` в памяти.
- **L3.** Повсюду повторный `SELECT * ... WHERE id=?` после каждого UPDATE (parser.py) — мелочь, можно вернуть обновлённую строку одним запросом.

- Per-request соединение в API (`deps.py:46`) — гонок на общем соединении больше нет.
- NO_PROXY-гейт: аккаунт без активного прокси не коннектится по реальному IP (`parsing_tasks.py:107, 694-698`).

- Анти-бан рандомные задержки между действиями и источниками.
- structlog с контекстом, AES-256-GCM для сессий.
