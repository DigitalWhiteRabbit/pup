> filesystem-брокер + воркер), read-only прозвон API. **Код не менялся.**
> Покрытие: 29 разделов (все экраны навигации + скрытые API-only модули).
>
> Сквозной вывод: **движок задач реально работает** (парсер, рассылки, инвайтинг, прогрев,

# КЛАСТЕР 1 — Аккаунты, профили, прокси, мессенджер

## Аккаунты (accounts)

**Сейчас умеет:**

- CRUD + импорт: `POST /accounts`, `/bulk-import`, `/import-zip` (распаковка `.session`+`.json`, AES-шифрование, авто-детект страны по номеру, статус SPAM_BLOCKED из spamblock-поля), экспорт в ZIP. Список с фильтром статус/поиск + пагинация; клиентские фильтры (страна, прокси, премиум, тег).
- Реальный TG-чек: `POST /{id}/check-telegram`, `/bulk-check-telegram` (семафор, NO_PROXY-guard, обновление ACTIVE/DEAD/BANNED/INVALID, enrichment scam/fake/verified/restricted/bio/2FA, индекс «человечности» 0..4, активные сессии).
- Bulk-действия `/bulk-action`: check_status, assign_proxy, set_status, add/remove_tags, delete + аудит-лог.
- Доп-чеки: `/check-spamblock` (@SpamBot), `/diagnose` (capabilities), `/verify-membership`.
- Карточка: вкладки Обзор/Активность/Прокси/Чаты/Баны/Профиль/Сторис/2FA, «Здоровье», live-сессии, лог прогрева, заметки.

**Предложения (отсортированы по ценности):**

1. **Активировать заглушки bulk-бара (Спамблок / Профиль / Очистка)** — три кнопки сейчас `wipToast`, хотя одиночные endpoint'ы готовы. Добавить bulk-spamblock и bulk-apply-profile с семафором. Объём: M. Зависимости: account_profile. Риск: med — горячий путь Telethon, но переиспользует готовые функции.
2. **Авто-чек по расписанию + фоновая задача** — весь чек синхронный в HTTP (большой пул = таймаут/блок event-loop). Перенести bulk-check на Celery с прогрессом + beat «ночной чек ACTIVE». Объём: M. Зависимости: Celery. Риск: med — трогает движок задач.
3. **История здоровья (health timeline)** — `last_check`/humanity перезаписываются, истории нет. Таблица снапшотов (статус, humanity, restricted, sessions) для графика деградации пула. Объём: M. Зависимости: schema. Риск: low — только запись.
4. **Колонка + фильтр по «здоровью»** — данные (humanity_score, restricted/scam/fake) уже в ответе, но в списке нет бейджа/фильтра «без профиля»/«ограничен». Объём: S. Зависимости: нет. Риск: low — чисто UI.
5. **Серверная пагинация клиентских фильтров** — фильтры страна/прокси/премиум/тег применяются к уже загруженной странице → при пагинации неполная выборка. Перенести в SQL. Объём: M. Зависимости: нет. Риск: low.

## Карточка аккаунта / профиль (account_profile)

**Сейчас умеет:**

- Локальная карта профиля `GET/PATCH /{id}/profile` (валидация bio≤70, username regex, privacy enum), TG не трогается.
- AI-генерация личности `/profile/generate-identity` (Claude Haiku, учёт страны/пола/ниши, выборочная регенерация, bio+bio_ru) и `/profile/translate`.
- Аватар: `/profile/generate-avatar` (thispersondoesnotexist.com, per-workspace) + `GET /profile/avatar` (containment-guard).
- Применение в TG `/profile/apply` по частям (name/username/photo/privacy), паузы 1-3с, per-part результат, отметка applied_at/parts.

**Предложения (отсортированы по ценности):**

1. **Bulk-генерация и применение профилей по пулу** — всё строго одиночное; 100 аккаунтов вручную не оформить. Пакетная генерация личностей+аватаров+поэтапное применение с разнесением во времени. Объём: L. Зависимости: accounts, Celery. Риск: med-high — массовые MTProto-write = ban-risk.
2. **Альтернативный источник аватаров / загрузка своих** — единственный источник часто 403/синтетика палится дедупом. Upload своего файла + seed/несколько источников. Объём: S. Зависимости: нет. Риск: low.
3. **Предпросмотр «как увидят аккаунт»** — собрать аватар+имя+@username+bio+privacy в одном превью перед apply. Объём: S. Зависимости: нет. Риск: low.
4. **Снапшот/откат предыдущего профиля** — `apply` перетирает name/username без бэкапа. Хранить snapshot + кнопка «Откатить». Объём: M. Зависимости: нет. Риск: low.
5. **Pre-check занятости username** — сейчас занятый ловится только ошибкой уже в TG (тратит вызов на живом акке). Лёгкий CheckUsername до apply. Объём: S. Зависимости: client_pool. Риск: low-med.

## Прокси (proxies)

**Сейчас умеет:**

- CRUD + bulk-import (host:port[:user:pass]), bulk-delete с каскадной отвязкой, фильтры status/type/country.
- Проверка `/{id}/check`, `/check-all` — TCP-connect (latency_ms, ACTIVE/DEAD), последовательно с паузой.
- Авто-привязка `/auto-assign`: geo_matching / equal_distribution / fill_one_first, dry_run, max_per_proxy, CIS-матч, качество PERFECT/CLOSE/NO_MATCH.
- Статистика `/stats`: by_status/type, unchecked, expiring_soon, accounts_per_proxy.

**Предложения (отсортированы по ценности):**

1. **Настоящая проверка (не только TCP-connect)** — `_check_proxy_connectivity` лишь открывает сокет; не проверяет auth, SOCKS/HTTP-хендшейк, доступность Telegram DC. Битый пароль = ACTIVE 63ms, потом аккаунт не подключается. Туннель через python_socks + коннект к Telegram, вернуть видимый IP. Объём: M. Зависимости: python_socks (есть). Риск: med — сетевые вызовы, изолированы от движка.
2. **Гео/IP-обогащение** — UI карточки уже ждёт `metadata.geo` (city/region/isp), но backend его не пишет → точнее geo_matching. Объём: M. Зависимости: geoip-сервис, фича №1. Риск: low.
3. **Слежение за истечением + авто-EXPIRED** — `expires_at`/`expiring_soon` есть, но нет задачи перевода истёкших в EXPIRED. Beat + бейдж. Объём: S. Зависимости: Celery beat. Риск: low.
4. **Расширенные форматы импорта** — парсер понимает только host:port[:user:pass]; не парсит `socks5://user:pass@host:port` и rotation_url. Объём: S. Зависимости: нет. Риск: low.
5. **Перепривязка аккаунтов мёртвого прокси** — при DEAD прокси аккаунты молча без рабочего прокси. Кнопка «перепривязать» поверх auto-assign. Объём: M. Зависимости: accounts. Риск: med.

## Мессенджер (messenger)

**Сейчас умеет:**

- Просмотр диалогов `/telegram/{id}/dialogs` и сообщений `/{id}/messages/{peer}` (детект media_type, reply_to, sender), NO_PROXY-guard.
- Отправка `/{id}/send` (text, reply_to), mark-read, активные сессии. Два входа в UI (отдельный экран + вкладка «Чаты»), поиск, поллинг 5с, mute AI-агента по треду.

**Предложения (отсортированы по ценности):**

1. **Отправка медиа и просмотр вложений** — только текст; `media_type` детектится, но контент не качается, файл отправить нельзя. `download_media` + `send_file`. Главная функциональная дыра «настоящего» мессенджера. Объём: L. Зависимости: нет. Риск: med — горячий путь, MTProto на живых акках.
2. **Переиспользование клиента вместо reconnect на каждый запрос** — каждый dialogs/messages/send открывает новый клиент, грузит до 100 диалогов, дисконнектится (медленно + ban-risk при поллинге 5с). client_pool/StringSession + кеш entity. Объём: M. Зависимости: client_pool. Риск: med.
3. **Удаление/редактирование/пересылка** — нет ни delete, ни edit, ни forward; базовый набор для ручной модерации. Объём: M. Зависимости: нет. Риск: med.
4. **Глобальный поиск сообщений и resolve по @username** — поиск фильтрует только загруженные диалоги; нет серверного `get_messages(search=)` и старта нового диалога. Объём: M. Зависимости: нет. Риск: low-med.
5. **Realtime вместо поллинга 5с** — постоянные reconnect'ы; event-handler/long-poll или условные запросы. Объём: M (L при WS). Зависимости: фича №2. Риск: med.

# СКВОЗНЫЕ УЛУЧШЕНИЯ (касаются многих разделов)

1. **Систематический рассинхрон UI↔API — главная боль платформы.** Повторяется почти везде: режимы парсера, роли/типы каналов, поведения автоответчика (FORWARD/IGNORE), имена полей настроек, FormData-vs-JSON конвертера, мок-экран прогрева, отсутствующий экран warmup_scripts, отсутствующий `/ai-sales/knowledge-base`. Половина — готовый бэкенд, не подключённый к фронту. Решение: контрактный smoke-тест UI→эндпоинты + единый источник enum'ов (отдавать через `/system`).

2. **«Фантомные» config-поля кампаний.** Формы DM/broadcast/invite/boost/cloner собирают десятки настроек (active*hours, filter*_, exclude\__, distribution, gap_24h, posting_time, permissions, username_pattern), но воркеры читают лишь 3-4 поля. Оператор настраивает фильтры и анти-бан, которые не применяются — это опасно (мнимая защита). Нужен аудит «UI-поле → реально читается» + реализация или скрытие.

3. **Суточные лимиты per-account не персистентны между запусками.** Везде `effective_limit` считается на один прогон таска; повторный старт в тот же день не знает, сколько акк уже сделал (опора на накопительный sent_count без сброса по дате). Дыра в анти-бане при рестартах/нескольких кампаниях на один акк. Решение: таблица `account_daily_usage` с reset по дате, проверяемая всеми движками горячего пути.

4. **Дублирование Telethon-инфраструктуры + разнородный анти-бан.** `_build_proxy_kwargs`, `_connect_account`, NO_PROXY-guard, FloodWait/PeerFlood-хендлинг, ramp-up почти идентичны в 6+ файлах (accounts, telegram_client, dm/chat/invite/boost-воркеры) тремя способами (temp-file vs StringSession). Свести к одному `app/services/tg_runner.py` с единым `can_act(account, action_type)` поверх Настроек. Снизит риск расхождений и переподключений.

5. **Дублирующиеся пути ответа на ЛС.** Минимум три механизма пишут в ЛС: DM-секретарь ai_agent, auto_replier, ai_sales. Нет арбитра «кто отвечает на этот peer» → риск двойных ответов с одного аккаунта (спам-паттерн, бан). Ввести слой «DM-ownership» (один обработчик на пару account+peer).

6. **RAG-качество — единая точка отказа всех AI-разделов.** Промоутер, продажник и KB-чат опираются на keyword-LIKE поиск без эмбеддингов (колонка `embedding` есть, не используется). Апгрейд KB-поиска улучшит сразу все AI-разделы — самое узкое место качества.

7. **Нет планировщика SCHEDULED / drip / AUTO_MONITOR.** Статусы SCHEDULED и поля start_type/posting_time/schedule_config/natural_curve есть в DM/broadcast/invite/boost/stories/cloner, но beat-задачи, которая их поднимает, нет — ничего не стартует/повторяется само. Один общий beat-планировщик «отложенных и повторяющихся кампаний» закроет сразу 6 разделов.

8. **Неоднородность «непрерывности» воркеров.** ai_agent/ai_sales/arena — self-rescheduling (+ reaper); commenting/auto_replier/boost/stories/cloner — одноразовый проход на `start`. Для пользователя «ACTIVE» означает разное поведение. Унифицировать паттерн (reaper/self-reschedule) и зачистить мёртвый `ai_promoter_tasks.py`.

9. **Наблюдаемость.** Аудит-лог пишется только частью операций (bulk-action, zip-import, auto-assign, warmup start/stop); чек, спамблок, apply профиля, отправка сообщений в `tg_audit_logs` не попадают — нет единой истории действий по аккаунту для разбора банов. Единый декоратор аудита на горячий путь.
