# Парсер аудитории — технический справочник

Инструмент сбора аудитории из Telegram: по списку источников (чаты/каналы) собирает
пользователей одним из 8 режимов, применяет фильтры и складывает результат в «базу
аудитории» с дедупликацией.

**Код:**

- `app/api/v1/parser.py` — CRUD + управление задачами (REST API).
- `app/tasks/parsing_tasks.py` — Celery-воркер: 8 режимов, ротация аккаунтов, анти-бан.
- `app/api/v1/audiences.py` — просмотр/поиск/экспорт собранных аудиторий.
- `app/parser.py` — серверный рендер UI раздела.

Вся работа идёт в фоне (Celery-задача `pup_tg.parse_audience`), UI опрашивает статус.

---

## 1. Режимы парсинга

Канонический список — `VALID_MODES` в `parser.py`. Ровно 8 режимов:

| Режим           | Что собирает                     | Метод Telethon                      | Требования / ограничения                                                                      |
| --------------- | -------------------------------- | ----------------------------------- | --------------------------------------------------------------------------------------------- |
| `CHAT_MEMBERS`  | Всех участников группы           | `get_participants(aggressive=True)` | Список участников должен быть открыт; у больших супергрупп Telegram отдаёт не более ~10k      |
| `WRITERS`       | Авторов сообщений                | `iter_messages(limit=500)`          | Берёт авторов последних 500 сообщений                                                         |
| `COMMENTERS`    | Авторов комментариев к постам    | `iter_messages` + `reply_to`        | Нужен канал с обсуждением; сканирует последние 100 постов, до 50 комментов на пост            |
| `REACTIONS`     | Поставивших реакции              | `GetMessageReactionsListRequest`    | Последние 50 постов; анонимные/канальные реакции пропускаются (только `PeerUser`)             |
| `POLLS`         | Голосовавших в опросах           | `GetPollVotersRequest`              | Последние 100 сообщений; работает только для **неанонимных** опросов                          |
| `JOINERS`       | Недавно вступивших               | сервисные сообщения                 | Последние 1000 сообщений; ловит `ChatAddUser` и `JoinedByLink`                                |
| `TOPICS`        | Участников тем форума            | `GetForumTopicsRequest`             | Только для форум-супергрупп; по 200 сообщений на тему. Не форум → пустой результат            |
| `GLOBAL_SEARCH` | Пользователей по ключевым словам | `contacts.SearchRequest`            | Не привязан к источнику; ключевые слова из `filters.keywords` (до 10), боты отфильтровываются |

Удалённые аккаунты (`deleted`) исключаются во всех режимах.

---

## 2. Фильтры

Применяются к результату любого режима (`_apply_filters`):

| Поле фильтра   | Действие                                                                  |
| -------------- | ------------------------------------------------------------------------- |
| `has_avatar`   | оставить только с аватаркой                                               |
| `has_username` | оставить только с @username                                               |
| `no_bots`      | убрать ботов                                                              |
| `premium_only` | оставить только Telegram Premium                                          |
| `keywords`     | (только `GLOBAL_SEARCH`) список запросов: массив или строка через запятую |

## 3. Модель данных

**`tg_parsing_tasks`** — задача парсинга:
`id`, `name`, `mode`, `config` (JSON), `status`, `progress` (0–100), `total_found`,
`total_filtered`, `error_message`, `audience_id` (FK на выходную базу), `celery_task_id`,
`started_at`, `finished_at`, `created_at`, `updated_at`.

**`tg_audiences`** — база аудитории:
`id`, `name`, `description`, `source_type` (`PARSED|IMPORTED|MERGED|FILTERED`),
`total_count`, `unique_count`, `tags`, `metadata`, таймстемпы.

**`tg_audience_members`** — участник базы:
`id`, `audience_id` (FK, `ON DELETE CASCADE`), `tg_user_id`, `username`, `first_name`,
`last_name`, `phone`, `about`, `is_premium`, `is_bot`, `has_avatar`, `source_chat`,
`last_seen_at`, `country`, `ai_score`, `ai_category` (`HIGH|MEDIUM|LOW|IRRELEVANT`),
`metadata`, `created_at`.

Дедуп — на вставке: `INSERT OR IGNORE` по уникальности участника в рамках базы.
`total_found` = сырых найдено, `total_filtered` = реально вставлено (после фильтров и дедупа).

---

## 4. Жизненный цикл и статусы

`VALID_STATUSES`: `PENDING → RUNNING → (PAUSED ↔ RUNNING) → COMPLETED | FAILED | CANCELLED`.

- **PENDING** — создана, не запущена.
- **RUNNING** — выполняется воркером.
- **PAUSED** — кооперативно приостановлена (частичный результат сохранён).
- **COMPLETED** — завершена штатно.
- **FAILED** — ошибка (нет источников/аккаунтов, глобальный FloodWait, креш).
- **CANCELLED** — отменена пользователем (Celery-задача отзывается).

Пауза/отмена **кооперативные**: проверяются перед обработкой каждого источника, поэтому
останавливают «на границе источника», а не мгновенно посреди него. Частичные результаты
не теряются и база НЕ помечается COMPLETED.

## 5. API

Все эндпоинты требуют admin-токен (`AdminAuth`) и воркспейс (`?workspace=<id>`).
Префикс роутера парсера — `/api/v1/parser`.

| Метод    | Путь                        | Назначение                                                                       |
| -------- | --------------------------- | -------------------------------------------------------------------------------- |
| `GET`    | `/parser/tasks`             | список задач; `?status=`, `?limit=` (1–1000), `?offset=`                         |
| `GET`    | `/parser/tasks/{id}`        | задача + прогресс                                                                |
| `POST`   | `/parser/tasks`             | создать задачу (→ 201); `mode` ∉ VALID_MODES → 400                               |
| `POST`   | `/parser/tasks/{id}/start`  | PENDING/PAUSED → RUNNING + диспатч; движок офлайн → **503** (статус не меняется) |
| `POST`   | `/parser/tasks/{id}/pause`  | RUNNING → PAUSED (иначе 400)                                                     |
| `POST`   | `/parser/tasks/{id}/cancel` | не-терминальная → CANCELLED + revoke (терминальная → 400)                        |
| `DELETE` | `/parser/tasks/{id}`        | удалить (если RUNNING — отзывает Celery-задачу)                                  |

Аудитории (`/api/v1/audiences`):

| Метод  | Путь                      | Назначение                                                                                      |
| ------ | ------------------------- | ----------------------------------------------------------------------------------------------- |
| `GET`  | `/audiences/{id}/members` | участники; `?search=`, `?is_premium=`, `?has_username=`, `?ai_category=`, `?limit=`, `?offset=` |
| `POST` | `/audiences/{id}/export`  | выгрузка всех участников JSON-массивом (CSV формируется в UI)                                   |

**Тело создания задачи** (`POST /parser/tasks`):

```json
{
  "name": "Крипто-чаты RU",
  "mode": "CHAT_MEMBERS",
  "config": {
    "sources": ["@crypto_chat", "https://t.me/defi_group"],
    "filters": { "has_username": true, "no_bots": true, "premium_only": false },
    "threads": 3,
    "save_to_audience": "new",
    "audience_name": "Крипто-аудитория"
  }
}
```

`save_to_audience`: `"new"` (создать новую базу) или `id` существующей базы (проверяется,
нет → 404).

## 6. Архитектура выполнения

Воркер `_parse_audience_async` идёт по источникам последовательно:

1. **Выбор аккаунтов.** `_pick_accounts` берёт до `min(кол-во источников, 5)` аккаунтов
   со статусом `ACTIVE` **и назначенным прокси**. Аккаунты без прокси исключаются
   полностью — **NO_PROXY guard**: коннект с реального IP сервера запрещён.
2. **Ротация по источникам.** Каждый источник обрабатывается следующим живым аккаунтом
   (round-robin, `_select_account`) — нагрузка размазывается, риск бана ниже.
3. **Кэш клиентов.** Telethon-клиент на аккаунт подключается лениво один раз и
   переиспользуется между источниками; все закрываются в `finally`, временные
   session-файлы удаляются.
4. **Кооперативная остановка.** Перед каждым источником проверяется статус задачи в БД;
   `PAUSED`/`CANCELLED` → остановка с сохранением частичного результата.
5. **Вставка и прогресс.** Результат режима фильтруется, вставляется с дедупом, прогресс
   (0–100) и счётчики обновляются после каждого источника.

### Обработка ошибок (на источник)

| Ситуация                                              | Реакция                                                                |
| ----------------------------------------------------- | ---------------------------------------------------------------------- |
| Источник приватный/недоступен (`ChannelPrivateError`) | пропустить источник, продолжить                                        |
| `FloodWait ≤ 300с`                                    | подождать (wait+5, максимум 120с) и продолжить                         |
| `FloodWait > 300с`                                    | пометить аккаунт `FLOOD_WAIT`, выкинуть из ротации, продолжить другими |
| `AuthKeyUnregistered`                                 | пометить аккаунт `DEAD`, выкинуть из ротации                           |
| `UserDeactivatedBan`                                  | пометить аккаунт `BANNED`, выкинуть из ротации                         |
| Прочее                                                | залогировать, продолжить со следующего источника                       |
| Все аккаунты выбыли                                   | задача → `FAILED`                                                      |

### Анти-бан

Случайные паузы между источниками (2–5с) и внутри режимов (диапазоны зависят от режима,
напр. реакции — до 2–5с между постами). FloodWait обрабатывается, как выше.

## 7. Эксплуатация / частые вопросы

- **«No active accounts» / задача сразу FAILED** — нет ACTIVE-аккаунтов с прокси. Назначь
  прокси аккаунтам (раздел Прокси) — без прокси парсер аккаунт не использует.
- **Пустой результат у `TOPICS`** — источник не форум-супергруппа.
- **Пустой результат у `POLLS`/`REACTIONS`** — опрос анонимный либо реакции анонимные.
- **`COMMENTERS` ничего не нашёл** — у канала нет привязанного чата обсуждений.
- **Старт вернул 503** — фоновый воркер офлайн (`./scripts/dev-up.sh` локально; PM2 на
  проде). Задача остаётся в PENDING/PAUSED, ложного RUNNING нет.
- **Дубликаты не появляются** — один и тот же пользователь в рамках одной базы не
  вставляется повторно (`INSERT OR IGNORE`).
