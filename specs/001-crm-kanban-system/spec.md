# Feature Specification: CRM — Канбан-система управления проектами и задачами

**Feature Branch**: `001-crm-kanban-system`
**Created**: 2026-05-05
**Status**: Draft
**Input**: Командная CRM с канбан-доской, учётом времени в работе и Telegram-уведомлениями

---

## User Scenarios & Testing *(обязательно)*

### User Story 1 — Авторизация и управление учётными записями (Priority: P1)

Администратор системы создаёт учётные записи сотрудников. Пользователи входят по логину/паролю.
Администратор может деактивировать пользователей, которые больше не работают в компании.

**Why this priority**: Без авторизации никакой другой функциональностью пользоваться невозможно.

**Independent Test**: Первый ADMIN создаётся через CLI-сидер. Затем через `/admin/users` создаётся USER.
USER успешно входит. ADMIN деактивирует USER — USER больше не может войти.

**Acceptance Scenarios**:

1. **Given** пустая БД, **When** запускается CLI-сидер с логином/паролем, **Then** создаётся первый ADMIN-аккаунт
2. **Given** залогиненный ADMIN, **When** он открывает `/admin/users` и создаёт пользователя, **Then** новый USER может войти с указанными данными
3. **Given** ADMIN деактивировал USER, **When** USER пытается войти, **Then** система отказывает с понятным сообщением
4. **Given** залогиненный ADMIN, **When** он назначает роль ADMIN другому пользователю, **Then** тот получает доступ к `/admin/users`
5. **Given** незалогиненный пользователь, **When** он пытается открыть защищённую страницу, **Then** происходит редирект на форму входа

---

### User Story 2 — Создание проектов и управление участниками (Priority: P1)

Пользователь создаёт проект и становится его владельцем. Добавляет коллег по логину или email.
Участники получают доступ к канбан-доске проекта.

**Why this priority**: Проект — основная единица работы. Без него нет доски и задач.

**Independent Test**: USER создаёт проект, добавляет другого USER как участника. Второй USER видит
проект в своём списке и открывает доску.

**Acceptance Scenarios**:

1. **Given** залогиненный USER, **When** он создаёт проект с названием, **Then** проект появляется в его списке с тремя колонками по умолчанию ("Ожидает", "В работе", "Готово")
2. **Given** OWNER проекта, **When** он ищет пользователя по логину/email и добавляет его, **Then** тот видит проект в своём списке
3. **Given** USER не участник проекта, **When** он пытается открыть доску проекта, **Then** система возвращает ошибку доступа
4. **Given** ADMIN, **When** он открывает список проектов, **Then** видит все проекты в системе

---

### User Story 3 — Работа с канбан-доской: колонки и задачи (Priority: P1)

Участники проекта создают задачи, перемещают их между колонками drag&drop, настраивают колонки.
Все изменения отражаются мгновенно (оптимистичные обновления).

**Why this priority**: Это ядро продукта — основная ежедневная работа пользователей.

**Independent Test**: Пользователь создаёт задачу в колонке "Ожидает", перетаскивает в "В работе" —
задача появляется в новой колонке без перезагрузки страницы.

**Acceptance Scenarios**:

1. **Given** открытая доска, **When** участник создаёт задачу с названием, **Then** задача появляется в выбранной колонке
2. **Given** задача в колонке A, **When** пользователь перетаскивает её в колонку B, **Then** UI обновляется немедленно; если сервер возвращает ошибку — задача возвращается на место
3. **Given** пустая колонка, **When** пользователь переименовывает её, **Then** новое имя отображается на доске
4. **Given** пользователь, **When** он перетаскивает колонку на другое место, **Then** порядок колонок меняется немедленно с откатом при ошибке
5. **Given** OWNER, **When** он удаляет колонку с задачами, **Then** система блокирует удаление и требует сначала переместить все задачи вручную; удаление пустой колонки происходит без подтверждения

---

### User Story 4 — Учёт времени в работе (Priority: P2)

Когда задача находится в колонке "В работе" — система автоматически считает время.
На карточке задачи виден суммарный таймер и живой счётчик.

**Why this priority**: Ключевое отличие от Trello. Важно, но не блокирует базовую работу с доской.

**Independent Test**: Задача перемещается в "В работе" — на карточке появляется живой счётчик.
Задача перемещается в "Готово" — счётчик останавливается, суммарное время сохраняется.

**Acceptance Scenarios**:

1. **Given** задача перемещена в колонку "В работе", **When** проходит время, **Then** на карточке обновляется живой счётчик
2. **Given** задача в "В работе" с открытым интервалом, **When** её перемещают в другую колонку, **Then** интервал закрывается, время добавляется к суммарному
3. **Given** колонку переименовали в "В работе", **When** в ней есть задачи, **Then** для всех них открываются новые интервалы
4. **Given** колонку "В работе" переименовали в другое, **When** в ней есть задачи с открытыми интервалами, **Then** все интервалы закрываются
5. **Given** задача с несколькими завершёнными интервалами, **When** открывается карточка, **Then** показывается суммарное время всех интервалов

---

### User Story 5 — Комментарии и вложения к задачам (Priority: P2)

Участники проекта обсуждают задачи через комментарии и прикладывают файлы.
Автор может редактировать и удалять свои комментарии. Вложения удаляет автор или OWNER.

**Why this priority**: Обогащает карточку задачи контекстом, но не блокирует основную работу.

**Independent Test**: Участник добавляет комментарий к задаче — он виден другим участникам.
Загружает файл — он скачивается другим участником. Не-участник не может скачать файл.

**Acceptance Scenarios**:

1. **Given** участник проекта, **When** он оставляет комментарий к задаче, **Then** комментарий виден всем участникам с именем автора и датой
2. **Given** автор комментария, **When** он редактирует или удаляет свой комментарий, **Then** изменение применяется; другой пользователь не видит кнопок редактирования чужих комментариев
3. **Given** участник проекта, **When** он прикрепляет файл к задаче, **Then** файл доступен для скачивания участникам проекта
4. **Given** пользователь не участник проекта, **When** он пытается скачать файл по прямой ссылке, **Then** получает ошибку доступа
5. **Given** загрузивший файл или OWNER, **When** он удаляет вложение, **Then** файл удаляется с сервера и из БД

### User Story 6 — Уведомления in-app и Telegram (Priority: P2)

Пользователи получают уведомления о важных событиях внутри приложения (колокольчик) и в Telegram.
Telegram подключается через одноразовый код в боте системы.

**Why this priority**: Улучшает командную коммуникацию, но система работает без уведомлений.

**Independent Test**: Пользователь A назначает USER B на задачу — у B появляется непрочитанное
уведомление в колокольчике. После подключения Telegram — B получает то же уведомление в Telegram.

**Acceptance Scenarios**:

1. **Given** пользователь назначен на задачу другим участником, **When** это происходит, **Then** в колокольчике появляется уведомление; себе уведомление не создаётся
2. **Given** задача, на которую назначен пользователь, **When** другой участник добавляет комментарий, **Then** назначенный получает уведомление
3. **Given** пользователь, **When** он нажимает "Подключить Telegram", генерирует код, идёт в бота и отправляет `/start <код>`, **Then** его Telegram привязан и он получает уведомления в Telegram
4. **Given** подключённый Telegram, **When** пользователь отключает отдельный тип уведомлений, **Then** этот тип больше не отправляется в Telegram
5. **Given** список уведомлений, **When** пользователь кликает "отметить прочитанными", **Then** счётчик сбрасывается

### Edge Cases

- Что если при drag&drop сервер вернул ошибку — задача должна вернуться на прежнее место с toast-сообщением
- Что если удаляют колонку "В работе" — удаление заблокировано, пока в колонке есть задачи; пользователь переносит задачи → интервалы закрываются автоматически при выходе задач из "В работе"
- Что если два пользователя одновременно перемещают одну задачу — last-write-wins; пользователь видит актуальное состояние после отклика сервера
- Что если код привязки Telegram истёк — пользователь генерирует новый; старый более не действителен
- Что если деактивированный пользователь назначен на задачу — задача остаётся, в карточке показывается "(деактивирован)"
- Что если ADMIN закрыл диалог создания пользователя, не скопировав пароль — пароль недоступен; ADMIN должен сбросить пароль (создать новый через отдельное действие)
- Что если Telegram API недоступен при отправке уведомления — in-app уведомление создаётся в любом случае; Telegram-транспорт делает 3 retry с exponential backoff, затем тихо логирует ошибку
- Что если OWNER покидает проект — за рамками MVP

## Clarifications

### Session 2026-05-05

- Q: Что происходит при попытке удалить колонку, в которой есть задачи? → A: Удаление заблокировано — нельзя удалить колонку, пока в ней есть задачи
- Q: Требования к паролю и процесс выдачи временного пароля? → A: 12 символов, латинские буквы (A-Z, a-z) и цифры (0-9), генерируется криптографически случайно; показывается ADMIN один раз при создании (после — не восстанавливается)
- Q: Поведение при недоступности Telegram API? → A: In-app уведомление создаётся всегда; Telegram-сбой логируется, 3 retry с exponential backoff, после — тихий сбой
- Q: Структура пути хранения файлов на диске? → A: `uploads/{projectId}/{taskId}/{uuid}-{originalName}`
- Q: Срок жизни сессии? → A: Сессия живёт до явного logout (без maxAge). При деактивации пользователя админом все JWT-сессии мгновенно перестают работать: NextAuth callbacks.session проверяет User.isActive → если false, возвращает null (принудительный logout)

## Requirements *(обязательно)*

### Functional Requirements

**Авторизация и пользователи**

- **FR-001**: Система MUST создавать первого ADMIN через CLI-сидер (без UI)
- **FR-002**: ADMIN MUST иметь возможность создавать пользователей с логином, email и ролью через `/admin/users`; система MUST автоматически генерировать криптографически случайный пароль (12 символов, A-Z a-z 0-9) и показывать его ADMIN один раз — после закрытия диалога пароль недоступен
- **FR-002a**: Пароль пользователя MUST соответствовать требованиям: минимум 12 символов, латинские буквы и цифры (A-Z, a-z, 0-9)
- **FR-003**: Система MUST аутентифицировать пользователей по логину или email + пароль с защитой сессии от CSRF; JWT-сессия живёт до явного logout (без maxAge)
- **FR-004**: ADMIN MUST иметь возможность деактивировать пользователей; деактивированные не могут войти, но их данные сохраняются
- **FR-004a**: При деактивации пользователя все его активные JWT-сессии MUST мгновенно перестать работать; реализуется через проверку `User.isActive` в `callbacks.session` NextAuth — если `isActive === false`, сессия возвращает `null` (принудительный logout при следующем запросе)
- **FR-005**: ADMIN MUST иметь возможность назначать и снимать роль ADMIN у других пользователей
- **FR-036**: ADMIN MUST иметь возможность сбросить пароль пользователя — система генерирует новый криптографически случайный пароль и показывает его ADMIN один раз

**Проекты**

- **FR-006**: USER MUST иметь возможность создавать проекты; создатель автоматически становится OWNER
- **FR-007**: OWNER проекта MUST иметь возможность добавлять участников по логину или email
- **FR-008**: USER MUST видеть только проекты, где он OWNER или MEMBER
- **FR-009**: ADMIN MUST видеть все проекты системы

**Колонки**

- **FR-010**: При создании нового проекта система MUST автоматически создавать три колонки: "Ожидает", "В работе", "Готово"
- **FR-011**: Участники MUST иметь возможность создавать, переименовывать и удалять колонки; удаление колонки с задачами MUST быть заблокировано — пользователь обязан переместить все задачи вручную перед удалением
- **FR-012**: Участники MUST иметь возможность менять порядок колонок через drag&drop
- **FR-013**: Колонка с именем "В работе" (без учёта регистра) MUST активировать учёт времени для задач внутри неё

**Задачи**

- **FR-014**: Участники MUST иметь возможность создавать, редактировать и удалять задачи
- **FR-015**: Задача MUST содержать: название (обязательно), описание (опционально), assignee (опционально), дату создания
- **FR-016**: Участники MUST иметь возможность перемещать задачи между колонками и менять порядок внутри колонки через drag&drop с оптимистичным обновлением UI и откатом при ошибке
- **FR-017**: Каждое перемещение задачи между колонками MUST логироваться с полями: кто, когда, откуда, куда

**Учёт времени**

- **FR-018**: При попадании задачи в колонку "В работе" система MUST открывать новый временной интервал
- **FR-019**: При выходе задачи из колонки "В работе" (любым способом) система MUST закрывать открытый интервал
- **FR-020**: Переименование колонки в "В работе" MUST открывать интервалы для всех текущих задач в ней
- **FR-021**: Переименование колонки из "В работе" MUST закрывать все открытые интервалы задач в ней
- **FR-022**: На карточке задачи MUST отображаться суммарное время (все интервалы) и живой счётчик (если задача в работе)

**Комментарии**

- **FR-023**: Участники MUST иметь возможность добавлять комментарии к задачам
- **FR-024**: Автор MUST иметь возможность редактировать и удалять свои комментарии; редактирование чужих запрещено

**Вложения**

- **FR-025**: Участники MUST иметь возможность прикреплять файлы любого формата и размера к задачам
- **FR-026**: Скачивание вложений MUST быть доступно только участникам проекта — проверка прав при каждом запросе
- **FR-027**: Удалять вложение MUST иметь возможность загрузивший его или OWNER проекта

**Уведомления**

- **FR-028**: Система MUST генерировать in-app уведомления для 4 событий: назначение на задачу, комментарий к задаче где ты assignee/автор задачи, перемещение задачи где ты assignee, добавление в проект
- **FR-029**: Уведомления о собственных действиях MUST NOT создаваться
- **FR-030**: В шапке MUST отображаться счётчик непрочитанных уведомлений
- **FR-031**: Пользователь MUST иметь возможность отмечать уведомления прочитанными и переходить к связанной задаче

**Telegram**

- **FR-032**: Пользователь MUST иметь возможность привязать Telegram через одноразовый код с TTL 10 минут
- **FR-033**: После привязки те же 4 типа уведомлений MUST отправляться в Telegram; in-app уведомление MUST создаваться независимо от статуса Telegram API; при сбое Telegram-транспорт MUST делать 3 попытки с exponential backoff, после чего тихо логировать ошибку
- **FR-034**: Пользователь MUST иметь возможность отключать отдельные типы Telegram-уведомлений
- **FR-035**: Пользователь MUST иметь возможность полностью отвязать Telegram в настройках профиля

### Key Entities

- **User**: логин, email, хэш пароля, роль (ADMIN/USER), статус активности, Telegram chat_id (опционально), настройки Telegram-уведомлений
- **Project**: название, описание, ownerId, список участников
- **ProjectMember**: userId, projectId, роль в проекте (OWNER/MEMBER)
- **Column**: projectId, название, порядок (position)
- **Task**: projectId, columnId, название, описание, assigneeId, порядок в колонке, дата создания
- **TimeInterval**: taskId, startedAt, endedAt (null если открыт)
- **ColumnMoveLog**: taskId, movedByUserId, movedAt, fromColumnName, toColumnName
- **Comment**: taskId, authorId, текст, createdAt, updatedAt
- **Attachment**: taskId, uploadedByUserId, originalName, size, mimeType, storagePath (формат: `uploads/{projectId}/{taskId}/{uuid}-{originalName}`), uploadedAt
- **Notification**: recipientId, тип (ASSIGNED/COMMENTED/MOVED/PROJECT_ADDED), taskId (опционально), projectId (опционально), isRead, createdAt
- **TelegramLinkToken**: userId, token, expiresAt (TTL 10 мин)

## Success Criteria *(обязательно)*

### Measurable Outcomes

- **SC-001**: Участник проекта может создать задачу и переместить её drag&drop менее чем за 10 секунд
- **SC-002**: После перемещения задачи drag&drop UI отражает изменение мгновенно (до получения ответа сервера)
- **SC-003**: При ошибке сервера во время drag&drop задача возвращается на исходное место с понятным сообщением — всегда, без исключений
- **SC-004**: Живой счётчик времени на карточке задачи обновляется каждую секунду без перезагрузки страницы
- **SC-005**: Новый ADMIN может быть создан и войти в систему менее чем за 5 минут после первого запуска
- **SC-006**: Telegram-уведомление доставляется пользователю в течение 5 секунд после события
- **SC-007**: Скачивание файла недоступно для пользователей вне проекта — 100% случаев
- **SC-008**: Все списки (задачи, уведомления, пользователи) отображаются с skeleton-UI во время загрузки
- **SC-009**: Интерфейс корректно работает на устройствах от 320px до 2560px, включая drag&drop на тач-экранах

## Assumptions

- Технологический стек: Next.js (App Router), Prisma, NextAuth.js для сессий — выбор подтверждается на этапе планирования
- Telegram-бот реализован на webhook или polling — выбор на этапе планирования
- Одноразовый код привязки Telegram — криптографически случайная строка (UUID или аналог)
- Порядок колонок и задач хранится как целочисленное поле position; при перестановке пересчитывается для затронутых записей
- Загрузка файлов без лимита означает отсутствие ограничения на уровне приложения; дисковое место — ответственность инфраструктуры
- Файлы хранятся по пути `uploads/{projectId}/{taskId}/{uuid}-{originalName}`; UUID предотвращает коллизии при одинаковых именах файлов
- Живой счётчик времени реализуется на стороне клиента (JS interval) с начальным значением из сервера
- Конфликт одновременного drag&drop двух пользователей разрешается по принципу last-write-wins; блокировки в MVP не предусмотрены

## Out of Scope (MVP)

- Чек-листы внутри задач
- Дедлайны и метки/labels
